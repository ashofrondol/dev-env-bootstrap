1. 이게 뭔가요?

새 컴퓨터·새 프로젝트마다 반복하는 "개발환경 세팅"을 Makefile 하나로 자동화한 도구입니다. 2026년 표준 파이썬 스택(uv + Ruff + pytest)과 웹 스택(Prettier + ESLint + Vitest)을 설치하고, VS Code 설정(저장 시 자동 포맷·자동 lint 수정)까지 깔아줍니다.

① make setup
전역 도구 설치 (1회) → ② make project
프로젝트 생성 (매번) → ③ 코딩
저장하면 자동 정리 → ④ make test
테스트 실행

2. 빠른 시작 (OS별)

LINUX / MACOS

터미널에서 바로

cd dev-env-bootstrap
make setup      # 최초 1회
make project    # 새 프로젝트
make test       # 테스트

make가 기본 설치돼 있으므로 추가 준비가 필요 없습니다.

WINDOWS

관리자 PowerShell에서

cd dev-env-bootstrap
powershell -ExecutionPolicy Bypass `
  -File bootstrap.ps1

WSL2가 없으면 자동 설치 → 재부팅 → 로그인 시 자동 재개됩니다. 이미 있으면 곧바로 WSL 안에서 make setup이 실행됩니다. 이후 작업은 WSL 셸(우분투 터미널)에서 make project 등을 사용하세요.

⚠ Windows 주의사항 WSL2는 BIOS/UEFI에서 가상화가 켜져 있어야 합니다 (에러 0x80370102 = 가상화 꺼짐). 또한 /mnt/c(Windows 디스크) 경로는 느리므로, 실제 프로젝트는 WSL 홈(~/)에 두는 것이 성능상 유리합니다.

3. 명령어 레퍼런스

명령하는 일언제
make help사용 가능한 명령 + 감지된 OS 출력궁금할 때
make setupuv·파이썬·Ruff·pre-commit 설치 + VS Code 익스텐션 일괄 설치컴퓨터당 1회
make project파이썬 프로젝트 생성 — 파이썬 버전을 물어보고(미입력 시 기본값), uv init·pytest·테스트 템플릿·VS Code 설정을 깔아줌새 프로젝트마다
make project PROJ_LANG=webHTML/CSS/JS 프로젝트 생성 (Vitest·Prettier·ESLint)웹 프로젝트
make project PROJECT_NAME=foo이름을 지정해서 생성필요 시
make testTestCase.txt 기반 pytest + smoke_test.py 실행코드 수정 후
make report실행 프로파일 보고서 생성 — 시간·메모리·구간별·입력별 측정을 reports/<날짜>.md로 저장성능 확인·비교 시
ℹ 변수 이름은 PROJ_LANG 언어 선택 변수는 Unix 로케일 환경변수 LANG과의 충돌을 피하기 위해 PROJ_LANG을 사용합니다. make project LANG=web은 동작하지 않습니다.

4. 설정 바꾸기 (dev.config)

프로젝트 루트의 dev.config에서 기본값을 바꿀 수 있습니다. 커맨드라인 인자가 항상 우선합니다.

DEFAULT_PYTHON_VERSION = 3.13   # 파이썬 기본 버전
PROJ_LANG = python              # 기본 언어 (python | web)
TEST_FRAMEWORK = pytest         # 테스트 프레임워크
PROJECT_NAME = my-project       # 기본 프로젝트 이름
# 일회성 재정의 예시
make project PROJ_LANG=web PROJECT_NAME=my-site
make setup DEFAULT_PYTHON_VERSION=3.12

5. 테스트 사용법 (TestCase.txt)

생성된 프로젝트의 TestCase.txt에 한 줄에 한 케이스를 적으면 pytest 파라미터화 테스트로 자동 변환됩니다. 코드를 몰라도 케이스를 추가할 수 있습니다. 

# 형식: 함수명 인자들 => 기대값   (#으로 시작하면 주석)
add 2 3 => 5
add 0.1 0.2 => 0.3                        # 실수는 근사 비교(pytest.approx)
concat "hello world" "!" => "hello world!"
head [10, 20, 30] => 10
merge {"a": 1} {"b": 2} => {"a": 1, "b": 2}
  • tests/test_from_testcases.py가 파일을 읽어 케이스마다 테스트를 만듭니다. 실패하면 어떤 입력이 실패했는지 pytest가 정확히 알려줍니다 (테스트 ID가 add(2, 3) => 5 형태로 표시).
  • 프로젝트 루트에 smoke_test.py가 있으면 make test가 이어서 실행합니다. "일단 돌아가긴 하는가"를 빠르게 확인하는 용도입니다.
  • make test는 저장소 루트에서 실행하면 PROJECT_NAME 디렉터리로 이동해 실행하며, 테스트 실패를 종료코드로 전파합니다 (CI에서 게이트로 사용 가능).
ℹ 지원 자료형 인자와 기대값은 파이썬 리터럴(ast.literal_eval)로 해석됩니다 — 정수·실수·불리언(True/False)·None·문자열("..."/'...')·리스트·튜플·딕셔너리. 리터럴이 아닌 토큰은 문자열로 취급되고(upper hello => HELLO 가능), 따옴표/괄호 안의 공백과 =>는 구분자로 취급하지 않습니다. 기대값이 실수면 자동으로 근사 비교합니다.

자동 실행 보고서 (make report)

make report를 실행하면 테스트 결과 + 실행 시간/메모리 + 구간별 처리 시간 + 입력별 자원 변동이 담긴 보고서가 reports/<날짜>.md로 생성되고, reports/history.csv에 한 줄 요약이 누적되어 코드 변경에 따른 성능 추세를 추적할 수 있습니다.

  • 전체 실행: 대상 스크립트(기본 smoke_test.py)의 벽시계/CPU 시간, 최대 메모리
  • 특정 구문: 코드에서 from perf import span 후 with span("파싱"): 으로 감싸면 구간별 시간·메모리 증가가 표로 표시
  • 느린 함수 찾기: cProfile 누적시간 상위 10 (계측 오버헤드가 있어 상대 비교용)
  • 입력 변동 → 자원 변동: TestCase.txt의 케이스별로 반복 실행(중앙값)해 입력 크기에 따른 시간/메모리 변화를 표로 제공
⚠ 측정값 해석 수치는 환경 부하에 따라 흔들립니다. 절대값 하나보다 history.csv의 추세, 같은 보고서 안의 입력 간 상대 비교로 판단하세요. tracemalloc 기반 메모리는 파이썬 객체만 계산합니다(C 확장 제외).

6. 설치되는 도구와 이유

도구역할왜 이걸 쓰나
uv파이썬 설치·가상환경·패키지·CLI 도구 관리pip/venv/pyenv/pipx를 하나로 대체. pip 대비 8~10배(웜 캐시 시 80배+) 빠름. 2026년 신규 프로젝트 기본값
Ruff린터 + 포매터Flake8·Black·isort를 한 도구로 대체. 저장(Ctrl+S)할 때마다 자동으로 코드를 정리·수정 — "자동 리팩터링"의 핵심
Pylance타입체크·자동완성Ruff와 역할이 달라 둘 다 설치 (중복 린팅은 꺼둠)
pytest파이썬 테스트사실상 표준. TestCase.txt 파라미터화와 궁합 최고
pre-commit커밋 직전 자동 검사커밋할 때마다 Ruff가 검사·수정 (ruff-check → ruff-format 순서)
Prettier / ESLint / Vitest웹 포맷 / 린트 / 테스트역할 분담이 업계 표준. Vitest는 Jest보다 빠르고 신규 프로젝트 기본
Claude Code 익스텐션AI 코딩 도우미사이드바에서 인라인 diff 리뷰. 유료 구독 또는 API 키 필요

VS Code 동작: templates/vscode/settings.json이 프로젝트에 복사되어 저장 시 자동 포맷 + import 정렬 + 고칠 수 있는 lint 경고 자동 수정이 켜집니다.

7. AI(Claude Code)와 함께 쓰기

이 저장소에는 Claude Code(주로 Opus)에게 작업을 시킬 때 주입하는 지침서 docs/AI-WORKFLOW.md가 포함되어 있습니다. 프로젝트의 하드 규칙(Makefile TAB, LF 줄바꿈, PROJ_LANG, uv 전용, Ruff 훅 순서 등)과 검증 절차, 프롬프트 요령이 정리되어 있습니다.

주입하는 세 가지 방법

  1. 상시 적용(권장) — cp docs/AI-WORKFLOW.md CLAUDE.md 로 복사. Claude Code가 매 세션 자동으로 읽습니다.
  2. 세션 중 참조 — 프롬프트에 @docs/AI-WORKFLOW.md 를 포함.
  3. 스크립트/CI — claude -p 프롬프트 앞에 파일 내용을 붙임.

좋은 요청의 공식: 위치 + 목표 + 검증 기준

# 나쁜 예
"테스트 기능 더 좋게 해줘"

# 좋은 예
"TestCase.txt에 예외 기대 문법을 추가해줘.
 'div 1 0 => raises ZeroDivisionError' 가 통과해야 하고
 기존 리터럴 케이스(정수/실수/문자열/리스트)도 깨지면 안 돼.
 수정 후 uv run pytest -v 실행해서 결과 보여줘"
ℹ 핵심 원리 AI에게 스스로 확인할 수 있는 합격 기준(테스트, 빌드, 예시 입출력)을 주면 사람이 일일이 검수하는 대신 AI가 스스로 고치고-검사하고-반복합니다. 자세한 요령은 AI-WORKFLOW.md 부록과 Anthropic 공식 모범사례 참조.

8. 문제 해결

증상원인 / 해결
make: command not found (Windows)Windows 네이티브에서 실행함. bootstrap.ps1을 통해 WSL 안에서 실행하세요.
Makefile: missing separatorrecipe 줄 들여쓰기가 스페이스로 바뀜. 반드시 TAB이어야 합니다.
스크립트 실행 시 $'\r': command not found파일이 CRLF로 저장됨. LF로 변환하세요 (dos2unix 또는 에디터 설정).
WSL 설치 에러 0x80370102BIOS/UEFI에서 가상화(Intel VT-x / AMD-V)를 켜세요.
'code' CLI를 찾을 수 없습니다VS Code 명령팔레트(Ctrl+Shift+P) → "Shell Command: Install 'code' command in PATH" 실행 후 make setup 재시도. WSL에서는 VS Code로 WSL 폴더를 한 번 열면 자동 설치됩니다.
make project LANG=web이 무시됨변수 이름이 PROJ_LANG으로 변경되었습니다.
pre-commit 훅이 오래된 Ruff 버전을 씀Ruff는 릴리스가 잦습니다. 프로젝트에서 pre-commit autoupdate를 주기적으로 실행하세요.
bootstrap.ps1에서 Unexpected token 파스 에러파일이 UTF-8 BOM 없이 저장됨. Windows PowerShell 5.1은 BOM이 없으면 ANSI로 읽어 한국어 주석이 깨집니다. 반드시 UTF-8(BOM 포함)으로 저장하세요. (반대로 .sh 파일에는 BOM이 있으면 안 됩니다)
